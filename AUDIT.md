# 🐺 帷幕之地 — 全面项目审计

> 审计日期：2026-06-26
> 当前版本：v1.5.2
> 代码规模：~3900 行（服务端 ~2700 + 客户端 ~1200）

---

## 一、架构 & 代码质量

### ✅ 做得好的

| 维度 | 评价 |
|------|------|
| 模块拆分 | Game / NightResolver / BotManager / handlers 职责清晰 |
| 实时通信 | Socket.IO 事件驱动，按需推送，无轮询开销 |
| 角色体系 | 8 种角色各具特色，能力设计有创意（种狼感染、守卫重伤、猎人武器腐蚀） |
| 房屋位置 | 独特的空间系统，夜晚出门/访客计数增加策略深度 |
| 安全基础 | XSS 过滤、速率限制、密码哈希、房间信令验证已到位 |

### ⚠️ 需要优化

| 问题 | 严重度 | 位置 | 说明 |
|------|--------|------|------|
| **Game.js 过大** | 中 | `server/src/game/Game.js` (925行) | 包含了阶段管理、计时器、投票、状态序列化、回放等，建议拆分为 PhaseManager / VoteManager / StateSerializer |
| **无输入验证层** | 高 | `handlers.js` | `night:action` 的 action/target/ability 未做服务端白名单校验，恶意客户端可发送非法 action |
| **定时器泄漏风险** | 中 | `Game.js` | `_clearPhaseTimeout` 后 `_phaseTimeout` 置 null，但如果 `clearTimeout` 在定时器已触发后调用则无效。`_startNightStepTimer` 重新设之前应确保旧定时器已清除 |
| **JSON 文件并发写** | 高 | `UserManager.js` | 多房间同时结束时并发写 `users.json` / `replays.json` 可能数据损坏。高频使用时需要文件锁或换 SQLite |
| **单例 IO 引用** | 中 | `Game.js:setIO()` | 每次 reconnect 都 setIO，但 `broadcast()` 用 `this._io` — 如果 IO 实例换了而旧引用还在某处，广播可能丢失 |
| **无类型检查** | 低 | 全项目 | JS 无类型，`nightTarget` 有时是玩家 ID 有时是屋子 ID，语义混淆风险 |

## 二、安全审计

### 🔴 高危

| 漏洞 | 说明 |
|------|------|
| **夜晚行动无服务端校验** | 客户端发什么就存什么。狼人可以设 `nightTarget` 为自己（刀自己），预言家可以查验已死玩家。服务端应校验 action 合法性 |
| **JSON 注入** | 用户名/房间号在日志和广播中未转义。`roomCode` 由 `generateRoomCode()` 生成（白名单安全），但 `playerName` 虽然 sanitize 了特殊字符，JSON 序列化前没有 escape |
| **无 CSRF 保护** | REST API 无 CSRF token，虽然后端只读，但 `/api/status` 和 `/api/lobby` 可被任意页面 fetch（CORS `*`） |

### 🟡 中危

| 漏洞 | 说明 |
|------|------|
| **回放数据无脱敏** | `_saveReplay` 保存了所有玩家的真实名字和角色，JSON 明文存储可被服务器管理员直接查看 |
| **PM2 环境变量明文** | `deploy-to-server.js` 中 `USER_PASSWORD_SALT` 和 `ROOM_PASSWORD_SALT` 硬编码在启动命令里，建议用 `.env` 文件 |
| **无请求体大小限制** | Express `express.json()` 未设 `limit`，可被大 payload DoS |

## 三、玩法深度分析

### 当前角色矩阵

| 角色 | 阵营 | 能力深度 | 策略空间 |
|------|------|----------|----------|
| 狼人 | 狼 | 刀人+相认 | ★★☆ |
| 种狼 | 狼 | 变身/感染/刀人 | ★★★ |
| 预言家 | 好 | 查验 | ★★☆ |
| 毒巫 | 好 | 烈性毒+药水 | ★★★ |
| 药巫 | 好 | 万能药+毒药 | ★★★ |
| 守卫 | 好 | 守护+重伤 | ★★★ |
| 猎人 | 好 | 追踪+反击+白天开枪 | ★★★ |
| 村民 | 好 | 出门观察 | ★☆☆ |

### 🔴 玩法短板

#### 1. 村民太弱
村民只有"出门看人数"一个行动，几乎没有策略空间。经典狼人杀中村民靠**发言和投票推理**，但这个项目：
- AI 村民不会真正推理发言
- 真人村民缺乏信息武器
- 访客计数对新手意义不大

**建议**：村民可加"偷听"能力（20% 概率听到目标屋子的角色类型），或"结伴"（两村民一起去同一家则免疫狼人攻击）。

#### 2. 缺乏公共信息博弈
- 死亡玩家不公开角色（只在游戏结束公开）— 这让好人方失去推理线索
- 缺少"遗言"机制
- 白天讨论阶段太机械（轮流 30 秒，无交互）

**建议**：死亡时公开角色（标准狼人杀规则），添加遗言系统。

#### 3. 投票缺乏张力
- 平票无人出局（无PK/二次投票）
- 无警长/归票机制
- 投票门槛（>50%存活玩家）在少人时太容易触发

**建议**：平票时进入 PK 环节（二次投票），添加警长竞选。

#### 4. 缺少以下经典狼人杀元素
| 缺失功能 | 影响 |
|----------|------|
| **警长竞选** | 无归票权，投票无方向 |
| **遗言** | 死亡玩家无最后发言 |
| **白狼王** | 自爆带走一人，制造突然死亡 |
| **丘比特/情侣** | 第三方阵营，大幅增加博弈复杂度 |
| **盗贼** | 换身份牌，增加不确定性 |
| **吹笛者** | 魅惑胜利条件，多了一条暗线 |
| **新月事件** | 随机事件增加变数 |

## 四、UX/UI 差距

| 问题 | 严重度 | 说明 |
|------|--------|------|
| **无新手引导** | 高 | 8 种角色能力复杂，新玩家进入直接懵 |
| **无游戏历史** | 中 | 回放只存最近 5 场，无战绩排行榜 |
| **移动端体验差** | 中 | Capacitor 集成了但 UI 未做触摸优化 |
| **无音效反馈** | 低 | 夜晚各步骤切换无提示音 |
| **无房间内角色预览** | 低 | 等待阶段看不到本局有哪些角色 |
| **无断线提示优化** | 中 | 断线只是文字提示，无头像灰化/倒计时 |

## 五、运维 & 扩展性

| 问题 | 建议 |
|------|------|
| 单进程无法扩展 | 加 `@socket.io/redis-adapter` 支持多进程 |
| 无健康检查端点 | `/api/status` 太简单，应加内存/CPU/连接数 |
| 无日志系统 | `console.log` 散落各处，应用 winston/pino |
| 无灰度发布 | PM2 直接 restart，应支持 reload 零停机 |
| 部署脚本密码硬编码风险 | deploy-to-server.js 命令行传密码，建议用 SSH key |
